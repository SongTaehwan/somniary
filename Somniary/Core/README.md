## Core

### 목적
애플리케이션 아키텍처의 기본 계약(Contracts)과 추상화 정의

```
┌─────────────────────────────────────┐
│              Core                   │  ← 최상위 추상화
│        (아키텍처 패턴 인터페이스)         │
└─────────────────────────────────────┘
            ↑         ↑         ↑
            │         │         │
    ┌───────┴────┐ ┌──┴────┐ ┌──┴──────┐
    │Presentation│ │Domain │ │Common   │
    │  (Views)   │ │(Logic)│ │(Utils)  │
    └────────────┘ └───────┘ └─────────┘
```

### 원칙
1. Protocol/Interface만 포함 (구현체 금지)
2. 프로젝트 전반에서 사용되는 아키텍처 패턴 정의
3. 최소한의 의존성 유지 (Foundation 수준)
4. 모든 레이어에서 참조 가능하나, Core는 다른 레이어를 참조하지 않음

### 포함 대상
- ViewModel 아키텍처 패턴
- Effect/Side Effect 처리 패턴
- 상태 관리 추상화
- 이벤트 핸들링 패턴

### 제외 대상
- 구체적인 구현체 (→ 각 레이어에서 구현)
- 비즈니스 로직 (→ Domain)
- UI 컴포넌트 (→ Presentation)
- 유틸리티 함수 (→ Common/Helpers)

### 추가 기준
새로운 아키텍처 패턴 추가 시 다음을 확인:
- [ ] Protocol/Interface 형태로만 구성되는가? (구현 코드 없음)
- [ ] 3개 이상의 모듈/레이어에서 공통으로 사용되는가?
- [ ] 아키텍처 패턴을 정의하는가? (유틸리티 함수가 아님)
- [ ] 비즈니스 로직에 독립적인가? (특정 도메인에 종속 X)
- [ ] Foundation 외 외부 의존성이 없는가?
- [ ] 구현체들이 이 추상화를 채택할 때 일관된 구조를 강제하는가?
